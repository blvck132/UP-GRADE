1. System Overview

UP-GRADE is a web-based student study portal designed for Kenyatta University. It provides a centralized platform where students and lecturers can upload, search, and download academic materials including study notes, past papers, and course resources. The application features a modern, responsive interface with support for both light and dark modes.

Primary Objectives

  • Resource Management: Centralized storage and retrieval of academic materials
  • User Authentication: Secure login and signup for different user roles
  • Document Search: Advanced filtering by school, course, and unit
  • Document Management: Upload, preview, and download capabilities
  • User Experience: Intuitive interface with dark/light mode support

Target Users

  • Students: Search and download study materials
  • Lecturers: Upload and manage course materials
  • Administrators: Manage system content and user permissions

2. System Architecture

UP-GRADE follows a client-side architecture with localStorage-based persistence. The application is built using vanilla JavaScript, HTML5, and CSS3, without external dependencies.

Architecture Components

Component Technology Purpose
Frontend HTML5, CSS3, JavaScript User interface and interactions
State Management localStorage API Persistent data storage
Authentication JavaScript (Client-side) User login/signup validation
File Handling FileReader API, Blob API PDF upload, storage, and preview
UI Framework Custom CSS System Responsive design and theming

Data Flow

Users interact with the UI → Events trigger JavaScript functions → Functions manipulate DOM and localStorage → UI updates to reflect changes → All data persists across sessions via localStorage.

5. Database Structure

The application uses browser localStorage for data persistence. All data is stored as JSON strings with keys prefixed by "ev_" (EduVault). The database is seeded with sample data on first load.

Database Schema

Users Table

{ id: string, // Student ID name: string, // Full name email: string, // Email address password: string, // Password (plaintext in localStorage) role: string // "student" | "lecturer" | "admin" }

Documents Table

{ id: string, // Unique document ID title: string, // Document title school: string, // Faculty/School name course: string, // Course/Programme name unit: string, // Unit/Module name type: string, // "study" | "past" uploadedBy: string, // Uploader name date: string, // Upload date (YYYY-MM-DD) size: string, // File size (formatted) fileName: string, // Original file name fileData: string, // Base64 encoded PDF data year: string|null // Year (optional) }

Reference Tables

  • schools - Array of Faculty/School names
  • courses - Object mapping schools to course arrays
  • units - Object mapping courses to unit arrays

Storage Limits

localStorage typically supports 5-10MB per domain. With base64 encoding overhead, actual file storage capacity is approximately 3-6MB per file across all documents.

⚠️ Warning: localStorage is not encrypted. For production use, implement server-side storage with authentication.

6. Authentication System

Login Flow

The login process validates user credentials against stored user records. Credentials are checked using simple string comparison (note: not suitable for production without encryption).

Function: attemptLogin()
Purpose: Validate login credentials and authenticate user
Parameters: None (reads from DOM elements)
Returns: void
Errors Handled: Empty fields, invalid credentials, non-existent user

Signup Flow

New users can create accounts with name, Student ID, email, and password. The system checks for duplicate Student IDs and validates email format.

Function: attemptSignup()
Purpose: Create new user account
Parameters: None (reads from DOM elements)
Returns: void
Validation: Name, ID, Email format, Password length (minimum 6)
Errors Handled: Empty fields, duplicate ID, invalid email

Session Management

  • currentUser - Global variable storing logged-in user object
  • logout() - Clears session and returns to login page
  • Role-based UI visibility controlled via JavaScript

7. Search & Filter System

Search Process

Users select School → Course → Unit, then click "Search Materials". The system filters the documents database and displays matching results in a grid layout.

Function: searchMaterials()
Purpose: Find documents matching selected criteria
Filter Logic: Exact match on school, course, and unit
Output: Grid of document cards with preview/download buttons

Cascading Selects

  • onSchoolChange() - Populates course selector when school is selected
  • onCourseChange() - Populates unit selector when course is selected
  • Empty options reset dependent selectors

Search Results Display

Results are displayed as document cards containing:

  • Document type badge (Study Material/Past Paper)
  • Document title
  • Unit, date, and year
  • File size
  • Preview and Download buttons
Function: clearSearch()
Purpose: Reset all search filters and clear results
Action: Resets selectors and clears results div

8. Upload Management

Upload Process

Users access the Upload tab (only available for Lecturers and Admins), select a PDF file, categorize it, and provide metadata. The system validates the file and stores it as base64 in localStorage.

File Upload Functions

Function: handleDragOver(e)
Purpose: Handle drag-over event for drop zone
Action: Adds visual feedback to drop zone
Function: handleDrop(e)
Purpose: Process dropped files
Action: Calls processFile() with dropped file
Function: handleFileSelect(e)
Purpose: Handle file input selection
Action: Calls processFile() with selected file
Function: processFile(file)
Purpose: Validate selected file
Validation:
  • Must be PDF (application/pdf)
  • Maximum size: 10MB
Output: Displays file preview

Upload Validation

Field Required Validation
PDF File ✓ PDF only, max 10MB
School ✓ Must be selected
Course ✓ Must be selected
Unit ✓ Must be selected
Title ✓ Cannot be empty
Year ✗ Optional
Function: uploadDocument()
Purpose: Upload validated PDF to database
Process:
  1. Validate all required fields
  2. Convert file to base64 using FileReader
  3. Show progress bar during "upload" animation
  4. Create document object and store in DB
  5. Reset form and refresh UI

9. PDF Preview & Download

Preview Modal

Users can preview documents before downloading. The preview opens in a modal window using an iframe to display PDFs.

Function: previewDoc(docId)
Purpose: Open PDF preview in modal
Process:
  1. Fetch document from database
  2. Set modal title to document title
  3. Load PDF into iframe
  4. Display modal overlay
  5. Show success toast
Function: closePreview()
Purpose: Close preview modal
Action: Hides modal and clears iframe
Function: downloadFromPreview()
Purpose: Download from preview window
Action: Calls downloadDoc() with current preview document

Download Functionality

The download system handles both actual uploaded files and sample documents. For actual files, the base64 data is used directly. For sample documents, a minimal PDF is generated on-the-fly.

Function: downloadDoc(docId)
Purpose: Download document to user's device
Process:
  1. Find document in database
  2. If fileData exists, use it directly
  3. If no fileData, generate sample PDF
  4. Create blob and download link
  5. Trigger download
  6. Clean up object URL

Keyboard Shortcuts

  • ESC - Close preview modal
⚠️ Browser Limitation: Some browsers may not fully render PDFs in iframe. Users should download files for full functionality.

10. Theme System

Theme Architecture

The application supports two themes: Dark (default for app pages) and Light (for login page and optional light mode). Theme preference is stored in localStorage and persists across sessions.

CSS Variables

Themes are implemented using CSS custom properties (variables) defined in :root. The body.dark-mode and body.app-light classes override these variables.

/* Light Theme (Default) */ :root { --ink: #0d0d0d; /* Text color */ --paper: #f5f0e8; /* Background */ --cream: #ece6d8; /* Secondary bg */ --gold: #c8922a; /* Accent */ --gold-light: #e8b84b; /* Light accent */ --rust: #b84c1e; /* Error color */ --forest: #1a4a2e; /* Primary (green) */ --forest-light: #2a6b42; /* Light primary */ } /* Dark Theme */ body.dark-mode { --ink: #f5f0e8; /* Light text */ --paper: #1a1a2e; /* Dark bg */ --cream: #2a2a3e; /* Secondary dark bg */ --forest: #5a9e6f; /* Brighter green */ }
Function: toggleTheme()
Purpose: Switch between light and dark modes
Process:
  1. Determine current theme
  2. Switch to opposite theme
  3. Apply theme to DOM
  4. Save preference to localStorage
  5. Update button label
Function: initializeTheme()
Purpose: Load theme on page load
Action: Retrieves saved theme preference and applies it

Theme Button

  • Located in navbar after login
  • Shows "Light Mode" in dark theme (moon icon)
  • Shows "Dark Mode" in light theme (sun icon)
  • Tooltip provides additional context

12. Installation & Setup

System Requirements

  • Modern web browser (Chrome, Firefox, Safari, Edge)
  • JavaScript enabled
  • localStorage support (all modern browsers)

Installation Steps

  1. Place all files in a web-accessible directory
  2. Open index.html in a web browser
  3. Database will auto-seed on first load
  4. No server setup required

Initial Setup

On first load, the application automatically seeds the database with:

  • 8 Kenyatta University Faculties
  • Multiple courses per faculty
  • Course units and modules
  • 5 Sample documents

Creating Test Accounts

Use the Sign Up feature to create test accounts. Available roles:

  • Student: Can search and download (assigned by default)
  • Lecturer: Can search, upload, and manage content
  • Admin: Full system access

To change user roles, modify the role field in the users array in localStorage, or update in attemptSignup() function.

Production Deployment Recommendations

⚠️ Important: This application is designed as a proof-of-concept. For production deployment:
  • Implement Server Backend: Use Node.js, PHP, Python, or similar for secure data storage
  • Add SSL/HTTPS: Encrypt all data in transit
  • Secure Authentication: Implement proper password hashing (bcrypt, Argon2)
  • Database Provider: Use MongoDB, PostgreSQL, MySQL, or similar
  • File Storage: Use cloud storage (AWS S3, Google Cloud, Azure)
  • User Permissions: Implement proper authorization checks on backend
  • Input Validation: Validate all inputs server-side
  • Security Headers: Add CORS, CSP, X-Frame-Options headers
  • Audit Logging: Log all user actions and file access

Troubleshooting

Storage Full Error

Solution: Clear browser cache or localStorage. Open DevTools → Application → clear Storage.

PDF Not Displaying

Solution: Some browsers have limited PDF viewer support. Download the file instead.

Dark Mode Not Persisting

Solution: Check if localStorage is enabled. It may be disabled in private/incognito mode.

Login Not Working

Solution: Ensure JavaScript is enabled. Check browser console for errors.
